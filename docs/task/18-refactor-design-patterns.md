# Task 18: 리팩토링 - 디자인 패턴 적용

## 목적
현재 코드베이스를 분석한 결과, 가독성을 높이고 향후 확장 시 유지보수를 용이하게 하기 위해 주요 모듈에 적절한 디자인 패턴을 적용합니다.

## 세부 작업 내역

### 1. UI 컴포넌트 동적 렌더링 (전략 패턴)
- **대상**: `src/lib/components/message-bubble.svelte`, `message-list.svelte`
- **내용**: 
  - 다양한 메시지 타입(텍스트, 이미지, 시스템 메시지 등)에 대응하기 위해 단일 `if/else` 분기문을 제거합니다.
  - 타입별 전략(렌더링 컴포넌트)을 매핑하여 Svelte `<svelte:component>` 또는 Svelte 5 Snippet 기반으로 동적 렌더링을 구현합니다.

### 2. 서버 비즈니스 로직 캡슐화 (퍼사드 패턴)
- **대상**: `src/lib/server/rooms/...`, `src/lib/socket.ts`
- **내용**: 
  - `createOneToOneRoom`과 같은 복잡한 트랜잭션, DB 접근, 고유키 생성 로직을 묶는 상위 퍼사드 클래스(`RoomFacade` 또는 `ChatService`)를 도입합니다.
  - API 라우트 및 소켓 핸들러가 내부 복잡성을 모르고 단일 메서드만 호출하도록 구조화합니다.

### 3. 일관성 있는 객체 생성 로직 도입 (팩토리 패턴)
- **대상**: `src/lib/types/chat.ts` (`toMessage` 함수)
- **내용**: 
  - 원시 소켓 데이터를 도메인 모델(ChatMessage)로 변환하는 작업을 `MessageFactory` 클래스로 캡슐화합니다.
  - 향후 다양한 타입의 메시지가 들어오더라도 생성 로직을 한 곳에서 관리할 수 있게 구조를 개편합니다.

### 4. 상태 동기화 결합도 낮추기 (옵저버/Pub-Sub 패턴)
- **대상**: 소켓 클라이언트 이벤트 핸들러 및 전역 상태(Store)
- **내용**: 
  - 소켓 이벤트 수신 시 직접 UI 컴포넌트를 변경하지 않고 이벤트를 Dispatch(발행)만 하도록 처리합니다.
  - 각기 다른 모듈이 이를 Subscribe(구독)하여 자신의 상태만 독립적으로 관리하도록 해 가독성을 개선하고 예측 가능하게 만듭니다.

## 테스트 및 검증
- 기존 E2E 테스트(`tc-01` ~ `tc-12`) 및 단위 테스트가 손상 없이 정상 동작하는지 확인합니다.
- 채팅방 생성, 메시지 송수신 흐름이 완벽하게 기존과 동일하게 동작하는지 검증합니다.
